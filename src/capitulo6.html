DDD Importa Hoje: Uma Entrevista com Eric Evans

InfoQ.com entrevista o fundador do Domain Driven Design, Eric Evans, para colocar o Domain Driven Design em um contexto moderno:


Por que o DDD é tão importante hoje quanto sempre?

Fundamentalmente, o DDD é o princípio de que devemos nos concentrar nas questões profundas do domínio em que nossos usuários estão envolvidos, que a melhor parte de nossas mentes deve ser dedicada a entender esse domínio e colaborar com especialistas nesse domínio para combatê-lo uma forma conceitual que podemos usar para construir software poderoso e flexível.

Este é um princípio que não sai de moda. Aplica-se sempre que estivermos operando em um domínio complexo e intrincado.

A tendência de longo prazo é aplicar software a problemas cada vez mais complexos, cada vez mais profundos no coração desses negócios. Parece-me que essa tendência foi interrompida por alguns anos, à medida que a web estourou sobre nós. A atenção foi desviada da lógica rica e das soluções profundas, porque havia muito valor em apenas colocar os dados na web, juntamente com um comportamento muito simples. Havia muito disso para fazer, e apenas fazer coisas simples na web foi difícil por um tempo, então isso absorveu todo o esforço de desenvolvimento.

Mas agora esse nível básico de uso da web foi amplamente assimilado e os projetos estão começando a ficar mais ambiciosos novamente em relação à lógica de negócios.

Muito recentemente, as plataformas de desenvolvimento web começaram a amadurecer o suficiente para tornar o desenvolvimento web produtivo o suficiente para DDD, e há uma série de tendências positivas. Por exemplo, SOA, quando bem usado, nos fornece uma maneira muito útil de isolar o domínio.

Enquanto isso, os processos ágeis tiveram influência suficiente para que a maioria dos projetos agora tenha pelo menos uma intenção de iteração, trabalhando em estreita colaboração com parceiros de negócios, aplicando integração contínua e trabalhando em um ambiente de alta comunicação.

Portanto, o DDD parece ser cada vez mais importante no futuro próximo, e algumas bases parecem estar estabelecidas.

As plataformas de tecnologia (Java, .NET, Ruby, outras) estão em constante evolução. Como o Domain Driven Design se encaixa?

Na verdade, novas tecnologias e processos devem ser julgados se dão suporte às equipes para que se concentrem em seu domínio, em vez de distraí-las dele. O DDD não é específico para uma plataforma de tecnologia, mas algumas plataformas oferecem maneiras mais expressivas de criar lógica de negócios e algumas plataformas têm menos confusão. Em relação aos últimos, os últimos anos indicam uma direção esperançosa, principalmente após o terrível final dos anos 1990.

Java tem sido a escolha padrão dos últimos anos e, quanto à expressividade, é típico das linguagens orientadas a objetos. Quanto à desordem de distração, o idioma base não é tão ruim. Tem garbage collection, que, na prática, acaba sendo essencial. (Em contraste com o C++, que apenas exigia muita atenção aos detalhes de baixo nível.) A sintaxe Java é um pouco confusa, mas os objetos Java simples e antigos (POJOs) ainda podem ser lidos. E algumas das inovações de sintaxe do Java 5 ajudam na legibilidade.

Mas quando os frameworks J2EE foram lançados, eles enterraram totalmente essa expressividade básica sob montanhas de código de framework. Seguindo as primeiras convenções (como EJB home, get/set prefixed accessors para todas as variáveis, etc.) produziam objetos terríveis. As ferramentas eram tão complicadas que absorviam toda a capacidade das equipes de desenvolvimento apenas para fazer funcionar. E era tão difícil mudar objetos, uma vez que a enorme confusão de código gerado e XML tinha sido vomitada, que as pessoas simplesmente não os mudavam muito. Essa era uma plataforma que tornava quase impossível a modelagem de domínio eficaz.

Combine isso com o imperativo de produzir UIs da Web mediadas por http e html (que não foram projetados para esse fim) usando ferramentas bastante primitivas de primeira geração. Durante esse período, criar e manter uma interface de usuário decente tornou-se tão difícil que pouca atenção foi deixada para o design de funcionalidades internas complexas. Ironicamente, no exato momento em que a tecnologia de objetos assumiu o controle, a modelagem e o design sofisticados sofreram um forte golpe.

A situação foi semelhante na plataforma .Net, com algumas questões sendo tratadas um pouco melhor e outras um pouco piores.

Esse foi um período desanimador, mas as tendências mudaram nos últimos quatro anos. Primeiro, olhando para Java, houve uma confluência de uma nova sofisticação na comunidade sobre como usar frameworks seletivamente e uma variedade de novos frameworks (principalmente de código aberto) que estão melhorando de forma incremental. Frameworks como Hibernate e Spring tratam de trabalhos específicos que J2EE tentou resolver, mas de uma forma muito mais leve. Abordagens como AJAX, que tentam resolver o problema da interface do usuário, de maneira menos trabalhosa. E os projetos são muito mais inteligentes agora sobre escolher os elementos do J2EE que lhes dão valor e misturar alguns desses elementos mais novos. O termo POJO foi cunhado durante esta época.

O resultado é uma diminuição incremental, mas perceptível, no esforço técnico dos projetos e uma melhoria distinta no isolamento da lógica de negócios do resto do sistema para que possa ser escrito em termos de POJOs. Isso não produz automaticamente um design orientado ao domínio, mas o torna uma oportunidade realista.

Esse é o mundo Java. Então você tem os recém-chegados como Ruby. Ruby tem uma sintaxe muito expressiva e, neste nível básico, deve ser uma linguagem muito boa para DDD (embora eu ainda não tenha ouvido falar de uso real dele nesses tipos de aplicativos ainda). Rails gerou muita empolgação porque finalmente parece tornar a criação de UIs da Web tão fácil quanto as UIs eram no início dos anos 1990, antes da Web. No momento, esse recurso tem sido aplicado principalmente na construção de alguns dos vastos números de aplicativos da Web que não têm muita riqueza de domínio por trás deles, já que mesmo esses foram dolorosamente difíceis no passado. Mas minha esperança é que, à medida que a parte da implementação da interface do usuário do problema for reduzida, as pessoas vejam isso como uma oportunidade de concentrar mais sua atenção no domínio. Se o uso de Ruby começar a ir nessa direção, acho que poderia fornecer uma excelente plataforma para DDD. (Algumas peças de infraestrutura provavelmente teriam que ser preenchidas.)

Mais de ponta são os esforços na área de linguagens específicas de domínio (DSLs), que há muito acredito que poderiam ser o próximo grande passo para o DDD. Até hoje, ainda não temos uma ferramenta que realmente nos dê o que precisamos. Mas as pessoas estão experimentando mais do que nunca nessa área, e isso me deixa esperançoso.

No momento, até onde eu sei, a maioria das pessoas que tentam aplicar o DDD está trabalhando em Java ou .Net, com algumas em Smalltalk. Portanto, é a tendência positiva no mundo Java que está tendo efeito imediato.

O que está acontecendo na comunidade DDD desde que você escreveu seu livro?

Uma coisa que me empolga é quando as pessoas pegam os princípios sobre os quais falei em meu livro e os usam de maneiras que eu nunca esperava. Um exemplo é o uso do desenho estratégico na StatOil, empresa petrolífera nacional norueguesa. Os arquitetos de lá escreveram um relato de experiência sobre isso. (Você pode lê-lo em http://domaindrivendesign.org/articles/.)

Entre outras coisas, eles usaram o mapeamento de contexto e o aplicaram à avaliação de software de prateleira em decisões de construção versus compra.

Como um exemplo bem diferente, alguns de nós têm explorado alguns problemas desenvolvendo uma biblioteca de código Java de alguns objetos de domínio fundamentais necessários para muitos projetos. As pessoas podem conferir isso em:
http://timeandmoney.domainlanguage.com

Estamos explorando, por exemplo, até onde podemos levar a ideia de uma linguagem fluente e específica de domínio, enquanto ainda implementamos objetos em Java.

Muita coisa está acontecendo lá fora. Eu sempre aprecio quando as pessoas entram em contato comigo para me contar sobre o que estão fazendo.
